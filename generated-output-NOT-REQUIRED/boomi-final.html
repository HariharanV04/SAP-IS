API Overview

The SuccessFactors Employee Central to NAVEX integration is a comprehensive data synchronization solution that extracts employee demographic data from SuccessFactors Employee Central and delivers it securely to NAVEX (a compliance and ethics platform) via encrypted SFTP transfer.

Base URL Pattern: - SuccessFactors: https://{tenant}.successfactors.com/odata/v2/ - NAVEX SFTP: Configured via NAVEXSFTPDirectory process property - Internal SFTP: Configured via Archive path on SFTP server process property

Authentication Mechanisms: - SuccessFactors: OAuth 2.0 or Basic Authentication with API keys - SFTP Servers: SSH Key-based authentication - Email: SMTP authentication

Processing Modes: - Delta Processing: Incremental updates based on Last Successful Run Date (LSRD) - Full Processing: Complete dataset extraction for specified date ranges

General Response Format: - Input: SuccessFactors OData JSON responses - Output: CSV format with PGP encryption for external delivery

Endpoints

SuccessFactors Employee Central Endpoints

Employee Data Extraction

  • HTTP Method: GET
  • Endpoint Pattern: /CompoundEmployee
  • Purpose: Extract employee demographic and employment information
  • Authentication: OAuth 2.0 Bearer Token or Basic Auth

Query Parameters:

Parameter Type Required Description Business Purpose

$select String No Field selection for performance Optimizes API response size
$expand String No Related entity expansion Includes employment and personal information

Complex Filtering Examples:

Filter Type Expression Business Context

Territory Filter company_territory_code in ('US','CA','MX') Geographic region filtering
Delta Processing last_modified_on gt datetime'2024-01-01T00:00:00' Incremental data extraction

Picklist Values Extraction

  • HTTP Method: GET
  • Endpoint Pattern: /PicklistOption
  • Purpose: Retrieve reference data for code translation
  • Query Example: $filter=picklistId in ('employee-status','GlobalEmployeeClass','jobRelType')

File Delivery

  • Protocol: SFTP
  • Authentication: SSH Key-based
  • Directory: Configured via NAVEXSFTPDirectory process property
  • File Format: Encrypted CSV with PGP encryption

File Archiving

Current Boomi Flow Logic

Process Flow Overview

The EC372 - EC to NAVEX integration implements a sophisticated employee data synchronization process with the following key characteristics:

  1. Trigger Mechanism: Scheduled execution with configurable frequency
  2. Processing Modes: Delta (incremental) or Full (complete dataset) extraction
  3. Data Sources: SuccessFactors Employee Central API with picklist reference data
  4. Data Transformation: Complex field mappings with business rule applications
  5. Delivery Methods: Encrypted SFTP delivery to external vendor and internal archiving
  6. Error Handling: Comprehensive exception management with detailed reporting
  7. Notification System: Email notifications for success and failure scenarios

Step-by-Step Flow Description

Step 1: Process Initialization (Main Process)

Component Type: Start Event

Purpose: Initiates the employee data synchronization process with comprehensive error tracking setup

Configuration Details: - Trigger Type: Scheduled or manual execution - Error Tracking: Initializes comprehensive error reporting across all component types - Property Validation: Verifies all required process properties are configured

Data Flow: - Input: Process trigger event - Processing: Initialize error tracking properties across 7 categories (Business Rule, Cleanse, Decision, Route, TryCatch, Return, General) - Output: Initialized process with error tracking capabilities

Step 2: Delta vs Full Processing Decision

Component Type: Decision (Delta Run?)

Purpose: Determines whether to perform incremental (delta) or full data extraction

Configuration Details: - Decision Logic: Evaluates FileType process property - Branch Conditions: - True: Delta processing using Last Successful Run Date (LSRD) - False: Full processing using configured date ranges

Data Flow: - Input: Process properties configuration - Processing: Evaluates extraction mode based on FileType parameter - Output: Routes to appropriate date range configuration

Step 3: Dynamic Date Range Configuration

Component Type: Document Properties (SetLastSuccessRunDate)

Purpose: Configures dynamic date ranges for data extraction based on processing mode

Configuration Details: - Dynamic Properties Set: 5 properties configured - Dynamic Process Property - SDATE: Process start date calculation - Dynamic Process Property - EDATE: Process end date calculation
- Dynamic Process Property - LSRD: Last successful run date tracking - Additional date calculation properties for filtering

Data Flow: - Input: Process properties and current system date - Processing: Calculates appropriate date ranges for delta vs full extraction - Output: Configured date parameters for SuccessFactors queries

Step 4: Where Clause Construction

Component Type: Sub-process (Sub: Where Clause Dynamic Filter)

Purpose: Builds dynamic WHERE clause for SuccessFactors API queries based on multiple filter criteria

Configuration Details: - Filter Categories: - Global Person ID (GPID) filtering - Company territory code filtering
- Employee class filtering - Legal entity filtering - Date range filtering (delta vs full)

Business Logic Flow:

GPID Filtering Decision: - Evaluates if GPID filter is not blank - If present: Adds person_id_external in ('gpid1','gpid2') to WHERE clause

Territory Code Filtering: - Evaluates if company_territory_code is not blank - If present: Adds territory filtering to WHERE clause

Employee Class Filtering: - Evaluates if employee_class is not blank
- If present: Adds class-based filtering to WHERE clause

Legal Entity Filtering: - Evaluates if Legal Entity is not blank - If present: Adds legal entity filtering to WHERE clause

Date Range Processing: - For Delta Run: Adds last_modified_on gt datetime'LSRD' - For Full Run: Adds effective_end_date >= 'start_date'

Data Flow: - Input: Process properties with filter criteria - Processing: Dynamic WHERE clause construction with multiple conditional filters - Output: Complete OData filter string for SuccessFactors API

Step 5: Parallel Data Extraction

Component Type: Parallel Processing (5 branches)

Purpose: Executes multiple data extraction operations simultaneously for performance optimization

Branch Configuration: - Branch 1: Employee demographic data extraction - Branch 2: Picklist reference data extraction - Branch 3: Manager information lookup - Branch 4: Employment history data - Branch 5: Additional attribute data

Data Flow: - Input: Configured WHERE clause and extraction parameters - Processing: Parallel execution of SuccessFactors API calls - Output: Multiple data streams ready for transformation and combination

Step 6: Employee Data Extraction (Sub: Extract Data)

Component Type: Sub-process with SuccessFactors Connector

Purpose: Executes primary employee data extraction from SuccessFactors Employee Central

Configuration Details: - Connector Type: SuccessFactors OData V2 - Operation: GET /CompoundEmployee - Authentication: OAuth 2.0 or Basic Authentication - Query Construction: Dynamic filter application with field selection

Key Data Elements Retrieved:

Field Category Fields Business Purpose
Personal Information person_id_external, first_name, last_name, email_address Employee identification
Employment Data employment_status, employee_class, pay_grade, position_title Role and compensation
Location Information location, company_territory_code, region, sector Geographic and organizational
Management Hierarchy manager_id, manager_person_id_external, relationship_type Reporting structure
Employment Dates hire_date, termination_date, original_start_date Employment timeline
Union Information custom_string30, custom_string31 Union participation tracking

Error Handling: - Connection timeouts and retry logic - Data validation for required fields - Exception logging with detailed error context

Data Flow: - Input: Constructed WHERE clause and field selection criteria - Processing: SuccessFactors API query execution with error handling - Output: Raw employee data in SuccessFactors OData JSON format

Step 7: Picklist Reference Data (Sub: Query Picklist Values)

Component Type: Sub-process with SuccessFactors Connector

Purpose: Retrieves reference data for code translation and data enrichment

Configuration Details: - Endpoint: /PicklistOption - Picklist IDs Retrieved: - employee-status: Employment status codes and descriptions - GlobalEmployeeClass: Employee classification codes - jobRelType: Job relationship type codes

Data Flow: - Input: Picklist ID configuration - Processing: Multiple API calls to retrieve reference data - Output: Cached reference data for transformation lookups

Step 9: Data Validation and Split Processing

Purpose: Validates transformed data and prepares for output generation with error segregation

Configuration Details: - Split Component: SplitON Log_Item for error record separation - Validation Rules: Required field validation and data type checking - Error Categorization: Separates hard errors from processable records

Processing Logic: - Data Validation: Checks for required fields (GPID, Email, Employment Status) - Error Record Handling: Routes invalid records to error reporting - Success Record Processing: Continues valid records to output generation

Data Flow: - Input: Transformed employee records - Processing: Data validation with error/success separation - Output: Validated records ready for CSV generation and error records for reporting

Step 11: File Encryption and Security

Component Type: PGP Encryption

Purpose: Secures employee data files using PGP encryption for external delivery

Configuration Details: - Encryption Method: PGP/GPG encryption with public key - Key Management: NAVEX public key for recipient decryption - File Extension: .pgp appended to encrypted files - Security Level: Industry-standard encryption for sensitive HR data

Data Flow: - Input: CSV file with employee data - Processing: PGP encryption using NAVEX public key - Output: Encrypted .pgp file ready for SFTP delivery

Step 12: SFTP Delivery and Archiving (Sub: SFTP to Pepsi and NAVEX)

Component Type: Sub-process with Multiple SFTP Operations

Purpose: Delivers encrypted files to external vendor and archives internally

Vendor Delivery Process:

Decision: Send Files to Vendor? - Evaluates Send Output To Vendor ? process property - If true: Proceeds with NAVEX SFTP delivery - If false: Skips vendor delivery

NAVEX SFTP Configuration: - Directory: Set via NAVEXSFTPDirectory process property - Authentication: SSH key-based authentication - File Naming: Timestamp-based unique naming - Error Handling: Connection retry logic with exception reporting

Internal Archiving Process:

Decision: Send Files to PepsiCo SFTP? - Evaluates SendFilesToPepsoSFTP process property
- If true: Archives to internal SFTP server - If false: Skips internal archiving

Archive Configuration: - Directory: Set via Archive path on SFTP server process property - File Retention: Based on compliance requirements - Encryption: Optional encryption for archived files

Data Flow: - Input: Encrypted employee data files - Processing: Parallel SFTP delivery to vendor and internal archive - Output: Successful file delivery confirmations or error notifications

Step 13: Email Notification Processing

Component Type: Mail Connectors with Dynamic Configuration

Purpose: Sends detailed execution reports to configured stakeholders

Success Notification Configuration: - From Address: fromEmailAddressForSuccess process property - To Address: toEmailAddressForSuccess process property
- Subject: Dynamic subject with timestamp and record counts - Content: Process execution summary with key metrics

Error Notification Configuration: - From Address: fromEmailAddressForErrors process property - To Address: toEmailAddressForErrors process property - Subject: Error alert with failure details - Content: Comprehensive error report with troubleshooting information

Email Content Elements:

Notification Type Content Elements

Error Error descriptions, failed record counts, system recommendations

Data Flow: - Input: Process execution results and statistics - Processing: Dynamic email content generation with execution metrics - Output: Email notifications sent to appropriate stakeholders

Step 14: Control and Error Reporting (Process for Control and Error Report)

Component Type: Sub-process for Comprehensive Reporting

Purpose: Generates detailed control reports and error analysis for process monitoring

Control Report Generation: - Record Counts: Total processed, successful, and error record counts - Execution Metrics: Start time, end time, duration, and performance statistics
- File Information: Generated file names, sizes, and delivery status - System Information: Process version, execution environment, and resource usage

Error Report Generation: - Error Categorization: Groups errors by type (validation, transformation, connectivity) - Error Details: Specific error messages, affected records, and remediation suggestions - Trend Analysis: Error pattern identification for process improvement

Archive Decision Logic: - Decision: Archive file to SFTP? - Logic: Evaluates archiving requirements and delivers reports to configured locations

Data Flow: - Input: Complete process execution data and error records - Processing: Comprehensive report generation with statistical analysis - Output: Control reports and error analysis ready for stakeholder review

Component Type: End Event

Purpose: Completes the integration process with final status reporting

Configuration Details: - Completion Type: Success or failure based on execution results - Final Actions: Process property updates and resource cleanup - Status Logging: Final execution status with complete audit trail

Data Flow: - Input: All process execution results and delivery confirmations - Processing: Final status determination and cleanup operations - Output: Process completion with comprehensive execution log
